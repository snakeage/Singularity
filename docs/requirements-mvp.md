# Requirements — MVP

Статус: черновик для первой учебной версии.  
Связанные документы: [vision.md](./vision.md), [methodology.md](./methodology.md), [data-model.md](./data-model.md).

## Цель MVP

Дать автору возможность вести **одну мечту** по лестнице этапов: точка А → этапы → рубежи и практики → отметки → недельный обзор. Без аккаунтов, без сервера.

## Область

**Входит:**

1. Создать Мечту (название, зачем, образ результата, горизонт).
2. Описать Точку А.
3. Разбить мечту на 2–5 этапов с критериями перехода.
4. У активного этапа: рубежи + ежедневные/еженедельные практики.
5. Отмечать выполнение, видеть прогресс этапа и связь с мечтой.
6. Короткий еженедельный обзор.

**Не входит:** соцсеть, бейджи, AI-коуч, мобильное приложение, команды, мульти-мечты как основной сценарий (одна мечта в фокусе достаточно; технически можно хранить несколько, но UX оптимизирован под одну активную).

## User stories и критерии приёмки

### US-01 — Создать мечту

**Как** пользователь, **хочу** описать глобальную цель (что / зачем / образ результата / горизонт), **чтобы** у пути был явный ориентир.

**Acceptance:**

- Можно создать мечту с полями: title, why, outcomeVision, horizon.
- Можно указать контекст/ограничения среды (опционально на старте, желательно).
- После создания мечта отображается как карточка-ориентир.

### US-02 — Зафиксировать точку А

**Как** пользователь, **хочу** честно описать текущую реальность, **чтобы** не строить план из иллюзий.

**Acceptance:**

- У мечты есть блок Point A: skills, resources, constraints, freeform notes.
- Point A можно сохранить и позже просмотреть.
- Point A доступна рядом с мечтой (не спрятана в настройках).

### US-03 — Разложить путь на этапы

**Как** пользователь, **хочу** разбить мечту на 2–5 этапов с objective и критериями выхода, **чтобы** глобальная цель не казалась бесконечно далёкой.

**Acceptance:**

- Можно создать от 2 до 5 этапов, упорядоченных по порядку.
- У этапа есть: title, objective, exitCriteria.
- Можно назначить ровно один этап статусом `active`.
- На экране видна цепочка этапов и подсветка текущего.

### US-04 — Рубежи активного этапа

**Как** пользователь, **хочу** задать измеримые рубежи этапа, **чтобы** понимать, когда я «вырос».

**Acceptance:**

- В активном этапе можно добавить Milestone: title, successMetric, optional dueAt.
- Рубеж можно отметить выполненным.
- Прогресс этапа отражает долю закрытых рубежей (и/или явные exitCriteria).

### US-05 — Практики (закалка)

**Как** пользователь, **хочу** завести повторяемые практики этапа, **чтобы** двигаться маленькими шагами каждый день/неделю.

**Acceptance:**

- Можно создать Practice: title, frequency (`daily` | `weekly`), optional cue (когда/где), link to why-for-stage.
- На «Сегодня» видны практики, подлежащие выполнению (чеклист).
- Сессия таймера — на экране **Разучивание** (`/drill`); с Сегодня есть вход.
- Можно поставить CheckIn (частично / норма / сильно / skipped) за период.
- Streak, если показывается, не блокирует и не стыдит при пропуске.

### US-06 — Связь действия с мечтой

**Как** пользователь, **хочу** видеть, как сегодняшняя практика связана с этапом и мечтой, **чтобы** не терять смысл рутины.

**Acceptance:**

- На экране дня/этапа есть явная цепочка: Practice → Stage → Dream.
- Заголовок мечты и название активного этапа видны без лишних кликов.

### US-07 — WOOP / препятствия (минимум)

**Как** пользователь, **хочу** записать препятствия и план «если → то», **чтобы** заранее знать реакцию на срыв.

**Acceptance:**

- У мечты или активного этапа можно сохранить Obstacle и ImplementationIntention (if → then).
- Они доступны для просмотра в обзоре и на экране этапа.

### US-08 — Источники роста (минимум)

**Как** пользователь, **хочу** привязать к этапу книги/курсы/менторов, **чтобы** помнить про обучение, а не только про чеклист.

**Acceptance:**

- У этапа можно добавить GrowthSource: title, type, optional url/notes.
- Список источников виден на карточке этапа.

### US-09 — Еженедельный обзор

**Как** пользователь, **хочу** раз в неделю коротко подвести итоги, **чтобы** корректировать курс.

**Acceptance:**

- Можно создать Review за неделю: worked, blocked, nextChange.
- Есть вопрос про окна обучения: использовал ли свободное время на учёбу под этап (`learningUsed` / короткий текст).
- Показываются простые факты периода: число CheckIn, закрытые Milestone.
- Обзоры сохраняются в истории.

### US-09b — Фильтр «не отклоняться» (лёгкий)

**Как** пользователь, **хочу** при добавлении практики видеть вопрос «это двигает к мечте/этапу?», **чтобы** не расползаться в боковые квесты.

**Acceptance:**

- При создании Practice есть опциональное подтверждение/поле связи с этапом (`whyForStage`).
- На экране дня видна цепочка Practice → Stage → Dream (см. US-06).

### US-10 — Локальное хранение без регистрации

**Как** пользователь, **хочу** пользоваться приложением локально без аккаунта, **чтобы** быстро начать.

**Acceptance:**

- Данные сохраняются в браузере (или локальной БД) без обязательного логина.
- После перезагрузки страницы данные на месте.
- Есть способ экспортировать/импортировать JSON-бэкап персонажа (полная замена при импорте).

## Нефункциональные требования (MVP)

- UI на русском.
- Работает как веб-приложение на desktop-браузере; mobile-friendly желательно, но не цель дизайна.
- Учебный код: TypeScript, понятная структура, документация рядом с кодом.
- Стек (когда дойдём до кода): Next.js + TypeScript, локальное хранение.

## Критерии готовности MVP

- [ ] Все US-01…US-09 выполнены по acceptance.
- [ ] Можно пройти сценарий «новая мечта → точка А → этапы → практики недели → обзор» без ручного редактирования storage.
- [ ] Документация в `docs/` соответствует фактическому поведению (или обновлена вместе с кодом).
